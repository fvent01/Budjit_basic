Budjit — Developer Manual

Technical reference for developers working on the Budjit self-hosted family budgeting application.

Architecture Overview

Budjit is a custom MVC PHP application with no third-party framework. All HTTP requests enter through a single front controller (public/index.php), which boots the router, loads enabled plugins, and dispatches to the appropriate controller.

→

→

→

View / JSON

Key design decisions

DecisionRationale
No framework Keeps the dependency surface minimal; everything is traceable by reading a handful of core files.
Plugin architecture Each optional feature (Savings Goals, Debt Tracker, etc.) lives in its own folder under /plugins. Plugins register their own routes and hooks so the core is never polluted by optional features.
PDO + prepared statements All database access uses PDO with bound parameters — no raw string interpolation, preventing SQL injection by construction.
AJAX-first UI Dashboard widget reordering, inline editing, and category management all use fetch() calls to return JSON — no full page reloads for these interactions.
Python sidecars Compute-heavy parsing tasks (PDF pay stubs via pdfplumber, Excel fallback via openpyxl) run as separate Python processes called from PHP via exec().
Plaid integration Bank transaction sync uses the Plaid API with cursor-based incremental sync. Access tokens are stored encrypted with AES-256.

Local Setup

Requirements

RequirementVersion
PHP8.1+
MySQL / MariaDBMySQL 8+ or MariaDB 10.4+
ApacheAny recent version with mod_rewrite
Python (optional)3.9+ — only needed for PDF pay stub parsing and Excel fallback

Step-by-step

1. Place files — Drop the project into your web server root, e.g. C:\xampp\htdocs\budjit\

2. Create the database

-- In phpMyAdmin or MySQL shell:
CREATE DATABASE budjit CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
-- Then import config/schema.sql via phpMyAdmin Import tab

3. Configure — Edit config/config.php:

define('BASE_URL', 'http://localhost/budjit/public');
define('DB_HOST',  'localhost');
define('DB_NAME',  'budjit');
define('DB_USER',  'root');
define('DB_PASS',  '');

4. Enable mod_rewrite — In httpd.conf, ensure AllowOverride All is set for the htdocs directory, and mod_rewrite is loaded. Restart Apache.

5. Visit http://localhost/budjit/public and register the first admin account.

Router

File: core/router/Router.php

The router is a simple pattern-matching dispatcher. Routes are registered in public/index.php (and plugin plugin.php files) before dispatch() is called.

Route registration

$router->get('/budgets',         'BudgetController@index');
$router->post('/budgets',        'BudgetController@store');
$router->get('/budgets/{id}',   'BudgetController@show'); // {id} becomes a param
$router->any('/some-path',     'SomeController@handle'); // GET + POST

Dispatch flow

On each request: Router::dispatch() normalises the URI (stripping the BASE_URL prefix for sub-directory installs), tries an exact match, then falls back to regex matching for parameterised routes like /budgets/{id}. Unmatched routes call ErrorController@notFound with a 404 status.

Handler format

Handlers are always "ClassName@methodName". The router instantiates the class and calls the method, passing any route params as ordered arguments.

// Route: /budgets/{id}/edit
class BudgetController extends Controller {
    public function edit(int $id): void { // $id injected from route
        $budget = (new BudgetModel())->find($id);
        $this->render('budget/edit', ['budget' => $budget]);
    }
}

Database

File: core/database/Database.php

Database is a singleton PDO wrapper. The single connection is created on first call and reused for the request lifetime.

$pdo = Database::getInstance(); // PDO instance, always utf8mb4
PDO settingValue
ATTR_ERRMODEERRMODE_EXCEPTION — all DB errors throw PDOException
ATTR_DEFAULT_FETCH_MODEFETCH_ASSOC — rows are associative arrays
ATTR_EMULATE_PREPARESfalse — real prepared statements, no emulation
Charsetutf8mb4 with utf8mb4_unicode_ci collation
🔒

Never catch a PDOException just to swallow it silently. Always log via error_log() before re-throwing or returning a user-safe error.

Auth & Sessions

File: core/auth/Auth.php

All authentication state lives in the PHP session. The session is started once in public/index.php via Auth::startSession() before any output.

Session payload

KeyTypeDescription
user_idintPrimary key from users table
user_namestringFull name (first + last)
user_roleintRole ID: 1 = Admin, 2 = Parent, 3 = Viewer
logged_inbooltrue when authenticated
login_atintUnix timestamp of login
csrf_tokenstring64-char hex token for CSRF validation

Common usage in controllers

Auth::requireLogin();    // redirects to /auth/login if not logged in
Auth::requireAdmin();    // 403 if not admin
Auth::requireWriteAccess(); // 403 if Viewer role
Auth::verifyCsrf();      // 419 if CSRF token missing/wrong — call at top of every POST handler

$user = Auth::user();    // ['id', 'name', 'role'] or null
$id   = Auth::id();      // int|null

CSRF protection

Every HTML form must include the CSRF hidden field. Every POST handler must verify it before processing.

<!-- In view template -->
<?= Auth::csrfField() ?>

// In controller POST handler — first line after requireLogin/requireWriteAccess
Auth::verifyCsrf();

Base Model

File: core/database/Model.php

All models extend Model. Subclasses declare a $table property and get CRUD methods for free.

class BudgetModel extends Model {
    protected string $table = 'budgets';
    // all base methods available immediately
}

Available methods

MethodReturnsDescription
find(int $id)?arrayRow by primary key
findAll($col, $dir)arrayAll rows, ordered
findWhere(array $conditions)arrayRows matching all conditions
findOneWhere(array $conditions)?arrayFirst matching row
insert(array $data)intInserts row, returns new ID
update(int $id, array $data)boolUpdates row by PK

✅

All findWhere and insert/update calls go through prepared statements automatically. Never build SQL by concatenating user input.

Plugin System

File: core/plugins/PluginLoader.php

Plugins are self-contained PHP packages inside /plugins/{slug}/. Each has a plugin.php entry point that registers routes and hooks. They are loaded automatically on every request if enabled.

Installed plugins

💳 Financial Import
financial-import
Unified import pipeline. Ingests CSV/OFX bank statements and Excel files into expenses. Supersedes the older bank-import and excel-budget-import plugins.
🎯 Savings Goals
savings-goals
Track progress toward financial targets with contribution logging and visual progress bars.
🏋️ Debt Tracker
debt-tracker
Log debts, record payments, and track payoff progress with interest calculations.
🔁 Recurring Bills
recurring-bills
Track subscriptions and regular payments. Dashboard widget shows upcoming bills.
🗓️ Calendar Reminders
calendar-reminders
Set financial reminders tied to calendar dates. A cron job in storage/cron/send_reminders.php handles delivery.
🏦 Bank Import (legacy)
bank-import
Legacy Plaid-based bank sync. Routes now redirect to the financial-import plugin.

Creating a new plugin

// 1. Create folder: plugins/my-feature/
// 2. Create plugins/my-feature/plugin.php:

<?php
defined('BUDJIT') or die;

// Register routes
$router->get('/my-feature', 'MyFeatureController@index');

// Register a dashboard widget
PluginLoader::register('dashboard_widgets', function() {
    ob_start();
    require PLUGIN_PATH . '/my-feature/views/widget.php';
    return ob_get_clean();
});

// Register a sidebar nav link
PluginLoader::register('nav_items', function($items) {
    $items[] = [
        'label' => 'My Feature',
        'url'   => BASE_URL . '/my-feature',
        'icon'  => 'ti-star',  // Tabler icon class
        'match' => 'my-feature',
    ];
    return $items;
});

// Render a view inside the main layout

Views & Layouts

Views are plain PHP files in app/views/. The base controller's render() method captures the view into $content and injects it into the layout.

Layouts

LayoutUsed for
layouts/main.phpAll authenticated pages. Renders the sidebar, flash messages, and plugin nav items.
layouts/auth.phpLogin and register pages (no sidebar).

Naming convention

Views follow the pattern feature/action.php — e.g. budget/create.php, budget/edit.php, budget/show.php. The render call omits the .php extension.

Icons

Icons use Tabler Icons (webfont, loaded from CDN). Use class ti ti-{icon-name}. Browse icons at tabler.io/icons.

Frontend

There is no frontend build step. All CSS and JS are hand-written vanilla files.

FileDescription
public/assets/css/app.cssAll application styles. Uses CSS custom properties (--blue, --green, etc.) that are overridden per-user from Settings → Appearance.
public/assets/js/app.jsDashboard drag-to-reorder, inline editing, AJAX form handlers, toast notifications, and theme switching.

Theming

Dark/light mode is driven by the data-theme="dark" attribute on <html>. The selected theme is persisted in localStorage and applied before first paint to avoid flash-of-wrong-theme. Primary and accent colours are loaded server-side as CSS variable overrides in the <head> from per-user preferences.

Progressive Web App

A minimal service worker (public/sw.js) is registered, enabling installability on mobile browsers.

Roles & Permissions

RoleIDCapabilities
Admin 1 Full access including plugin management, user management, and all settings.
Parent / User 2 Full read/write access to financial data. Cannot manage plugins or other users.
Viewer 3 Read-only. Any controller action that creates/updates/deletes data must call Auth::requireWriteAccess() at the top, which returns 403 for Viewers.

Guard call order in controllers

public function store(): void {
    Auth::requireLogin();        // 1. must be logged in
    Auth::requireWriteAccess();  // 2. must not be Viewer
    Auth::verifyCsrf();         // 3. CSRF token valid
    // ... handle POST data
}

Before any production or network deployment: rotate PLAID_SECRET and PLAID_ENCRYPTION_KEY, set a strong DB_PASS, set APP_ENV='production', and enforce HTTPS.

Adding Features

Adding a core page (non-plugin)

1. Create app/controllers/MyController.php extending Controller.
2. Create app/models/MyModel.php extending Model with $table set.
3. Create view(s) in app/views/my-feature/.
4. Register routes in public/index.php before PluginLoader::boot().
5. Add a nav item in app/views/layouts/main.php if needed.

Adding a plugin feature

1. Create plugins/my-plugin/plugin.php — register routes and hooks.
2. Add controller and model files in app/controllers/ and app/models/ (the autoloader finds them there).
3. Create views in plugins/my-plugin/views/.
4. If new DB tables are needed, create plugins/my-plugin/schema.sql and import it.
5. Add the plugin slug to config/plugins.php.

💡

Plugin controllers and models live in app/controllers/ and app/models/ respectively — the autoloader searches those paths. Only the plugin.php entry point and views live inside /plugins/{slug}/.

Coding standards

  • Always call Auth::requireLogin() (and write-access / admin guards) as the first line of any controller method.
  • Call Auth::verifyCsrf() at the top of every POST handler.
  • Use prepared statements via the base Model methods. Never concatenate user input into SQL.
  • Escape all output with htmlspecialchars().
  • Use password_hash() / password_verify() for all passwords — never MD5/SHA1.
  • Log errors with error_log(); never display raw exception messages to users in production.
